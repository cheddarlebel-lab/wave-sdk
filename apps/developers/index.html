Read this first — who can open a door

The SDK sends the credential. It does not decide access. A door only opens for a member the site's access provider already knows.

The one rule that governs everything: a Wave credential resolves to a cardID, and the door's controller grants only if that cardID exists in the site's provider source (the club's live membership system) or its synced provider cache — and passes the membership's rules (schedule, dues, check-in limits). Your app cannot mint access for a user who isn't a member of the site. "Has the app" ≠ "gets in."

The two identities you'll deal with

Provider member id what your user types

The id a member has in the site's membership system (MindBody client id, barcode, etc.). Used once, at enrollment, to find the person.

cardID the door truth

The canonical door identity the controller validates. The device resolves the typed member id → cardID at enrollment and binds your credential to it. Every unlock is checked against this.

Two ways to make your users door-eligible

  1. They're already members of the site. Default case. The site runs its own provider (MindBody, ABC, Glofox, TwinOaks…); your user enrolls with the member id they already have, the device resolves it to their cardID, and you're done. Nothing to provision.
  2. Passport is the provider for the site. When there's no third-party membership system, the site can run a Passport-hosted roster (managed in the Passport dashboard). You provision your users into that roster first; the controller syncs them and their credentials then resolve normally. This is an operator/site setup step, arranged with Passport — not something the SDK does at runtime.
Building against a site you don't operate? Confirm with the site operator (or Passport) which provider the door uses and that your users exist in it before you ship. Enrollment for a non-member fails with not_found and no credential is issued — the SDK surfaces that, but it can't fix it.

Quickstart

Prove the whole unlock flow in your terminal, no door required.

  1. Get keys. Ask your agent to run wave_register_app, or use the keys from your Wave dashboard. You get a wave_pub_* (in-app) and a wave_test_* (testing).
  2. Exchange for a token. POST /partner-auth/token with your test key returns a short-lived session token.
  3. Simulate an unlock. POST /unlock-mock with a scenario, then read /unlock-stream — the exact result a real door would return.
  4. Wire the real thing. Use wave_scaffold web (runs today via Web Bluetooth) or a native SDK; embed your wave_pub_* key.

Try it live now — copy, paste, run

A public demo test key. It only drives the mock (synthetic events on an isolated demo site) — it cannot open a real door. You'll get a real granted back from the live gateway.

BASE="https://app.wavepassport.com/api"
TEST_KEY="wave_test_9097529120bb93a92d058d65007fb555"

TOKEN=$(curl -s "$BASE/partner-auth/token" \
  -H "Content-Type: application/json" -d "{\"key\":\"$TEST_KEY\"}" | jq -r .token)

curl -s "$BASE/unlock-mock" -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" -d '{"card_id":"10001","scenario":"granted"}'

curl -s "$BASE/unlock-stream" -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" -d '{"card_id":"10001"}'
# -> {"status":"granted","reason":"[mock] Access Granted"}

How the unlock works

Six steps, hidden behind one SDK call. The SDK owns the phone-side; the controller and provider own the decision.

  1. Scan for the reader — name SKBluTag / service 496B2C43-….
  2. Proximity gate on signal strength (RSSI), tunable per site.
  3. Write the credential — a one-time token bound to the member's cardID, write-without-response.
  4. Resolve — the on-door acceptor verifies the token and hands the member's real cardID to the controller (SICM).
  5. Decide — the controller validates that cardID against the site's provider source/cache and its membership rules, then grants or denies.
  6. Await & show the result over the cloud (up to 5s): "Access Granted" or a friendly denial reason.

Steps 4–5 are outside your app — they're why a valid token can still be denied (expired membership, outside hours, unknown member). Your app's job is steps 1–3 and rendering step 6.
